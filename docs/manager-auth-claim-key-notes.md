# Manager Auth Claim-Key Notes

Date: 2026-05-28

Meeting notes:

- New manager-auth users should get an `ozw_` parent key autogenerated on first login.
- The app should still provide a claim-key action for users who already had an existing `ozw_` key.
- The claim-key flow should let a logged-in manager user enter an existing `ozw_` key once and claim the agents owned by that key.
- If a user creates agents under the newly autogenerated key before remembering their old key, claiming the old key should migrate those newly created agents onto the claimed old parent key.
- After that migration, the temporary autogenerated parent key should be deleted or retired.
- The app should include a way for the user to view/copy their own parent `ozw_` key.
- From the app's point of view, a first-time manager-auth user is new until they claim an existing key.

Implications to design:

- First login should create:
  - `users` row from manager auth headers.
  - autogenerated active `api_keys` parent key for that user.
- Claim existing key should:
  - validate the submitted `ozw_` key.
  - ensure it is safe to claim.
  - move agents from the autogenerated key to the claimed key.
  - attach the claimed key to the manager user.
  - delete or revoke the autogenerated key after migration.
- Parent key reveal should be explicit user action, similar to agent key reveal.
- Need clear rules for conflicts, such as a submitted key already linked to another manager user.
